# LocalMuse 改进方案：面向建筑学图像快速检索

> 目标只有一个：**用最少的操作、最短的时间，找到合适的建筑参考图。**
> 本文档为修改计划，暂不改动任何代码。

---

## 一、现状诊断

### 1.1 当前系统在做什么

- 索引端：每张图提取 CLIP 语义向量（512-D）、三代草图/结构向量（legacy 64×64、sketch_fit、sketch_crop）、MiDaS 深度、YOLOv8 人体姿态、主色/调色板、EXIF、（可选）OCR，写入 6+ 个 FAISS 槽位。
- 检索端：多槽位并行召回 → 归一化 → 加权融合。
- 反馈端（L2–L4）：文件名词频生成"意图候选卡片"、hover/skip/expand/save/exclude 行为信号、每 5 次搜索弹出 pairwise 二选一、多样性提示条、softmax 自适应模态权重、L3 标注记录导出。

### 1.2 核心问题：系统为"研究采集"设计，而不是为"快速找图"设计

| 问题 | 具体表现 | 对找图速度的影响 |
|---|---|---|
| 反馈机制打断心流 | pairwise 弹窗强制二选一、diversity nudge、意图卡片挡在搜索前 | 每次搜索多 1–3 步操作 |
| 自适应权重不可预测 | softmax(T=1.5) + 动量 + EMA，权重被压向均匀分布，且用户看不懂为什么这次结果和上次不一样 | 结果不可复现，用户失去掌控感 |
| 无关模态拖累索引与检索 | 人体姿态（建筑图库几乎无人体）、EXIF 相机参数（图库多为渲染图/网图）、OCR 残留 | 索引慢、库体积大、维护复杂 |
| 打标逻辑无信息量 | 意图候选来自**文件名词频**——建筑图常见文件名是 `IMG_2034.jpg`、`final_v3.png`，词频毫无语义 | 意图卡片形同虚设 |
| 缺少最关键的检索方式 | 没有"以图搜图"——找参考图最自然的动作是"再给我几张像这样的" | 用户只能反复改文字描述 |
| 结构槽位三代冗余 | legacy sketch(4096-D) + sketch_fit(16384-D) + sketch_crop(16384-D) 并存，检索时互相回退 | 逻辑复杂、易出 bug |

## 二、改进方案总览

三条主线：**① 框架做减法 → ② 打标换成建筑领域标签体系 → ③ 检索改成"以图搜图 + 标签过滤 + 轻量反馈"闭环。**

```
旧流程：输入文字 → 意图卡片选择 → 搜索 → (弹窗/nudge打断) → 权重暗中漂移 → 结果变化难解释
新流程：输入文字/拖入图片 → 结果 + 标签面板 → 点"更多类似"/"排除" → 结果即时收敛 → 收藏
```

## 三、框架精简（删除清单）

### 3.1 直接删除的功能

| 删除项 | 理由 |
|---|---|
| **pose 槽位**（YOLOv8、pose_model.py、pose 索引/检索/权重） | 建筑图库中人体姿态无检索价值 |
| **depth 槽位**（MiDaS、depth_model.py） | 现状大多数库无有效深度向量，实际走的是 sketch 代理（`_depth_proxy_scores`）——等于变相的结构检索，直接并入结构槽即可；MiDaS 还是最重的索引依赖 |
| **OCR 残留**（ocr_model.py、`ocr_enabled` 设置、索引中的 OCR 提取） | 已从检索移除，索引端残留只增加耗时 |
| **EXIF 提取**（相机/ISO/快门/光圈/GPS） | 建筑参考图多为渲染图、书籍扫描、网图，EXIF 为空或无意义 |
| **pairwise 二选一弹窗**（前后端全套：overlay、`/api/pairwise/choice`、`should_trigger_pairwise` 等） | 强打断、样本随机（从结果里随机抽两张），产生的偏好数据质量低 |
| **hover 计时信号**（>3s hover 视为正反馈） | 噪声极大（用户可能只是在看细节或走神），且前端要维护计时器 |
| **skip 信号** | 同上，滚动速度≠不喜欢 |
| **diversity nudge 提示条** | 阈值拍脑袋（0.15），且"结果相似"往往正是用户想要的（收敛中） |
| **softmax 自适应权重**（`_adjust_weights`、EMA、动量、置信度混合） | 见 1.2；softmax 归一后权重趋向均匀，学习效果本身存疑；用显式反馈替代（见第五节） |
| **L3/L4 标注导出体系**（L3Record、annotation frames、intent trajectory 时间线） | 属研究数据采集，与"快速找图"无关；如论文需要可留一个极简的事件日志（append-only JSONL），从主链路解耦 |
| **legacy sketch 64×64 槽位** | 保留 fit/crop 双通道即可，用 Fill Info 补全旧库后删除回退逻辑 |
| **基于文件名词频的 IntentEngine** | 被第四节的标签体系整体替代 |

### 3.2 精简后的槽位

| 槽位 | 用途 | 说明 |
|---|---|---|
| `semantic` | 文字搜图 + **以图搜图** | CLIP 图像向量本来就在库里，以图搜图零额外索引成本 |
| `structure`（fit + crop 双通道） | 草图/构图检索 | 合并原 sketch_fit/sketch_crop，删除 legacy 回退 |
| `color` | 色彩氛围过滤 | 保留现有主色/调色板 |
| `tags`（新增，见第四节） | 过滤与导航 | 元数据过滤，不占 FAISS |

索引依赖从 CLIP+MiDaS+YOLO+OCR 减到只剩 CLIP + OpenCV Canny，索引速度和安装体积都会显著改善。

## 四、图像打标逻辑：建筑领域零样本标签体系

### 4.1 核心思路

利用已有的 CLIP 图像向量做**零样本分类**：预先把一组建筑领域标签编码成文本向量，索引时（或对旧库离线批量跑一次）与图像向量求相似度，超过阈值即打标。**不需要任何新模型、不需要训练**，对旧库补打标签只是矩阵乘法，几万张图秒级完成。

### 4.2 标签体系（建筑学专用，分维度互斥/多选）

| 维度 | 候选标签（示例，可按个人图库调整） | 选择方式 |
|---|---|---|
| **图像类型** | 实景照片 / 渲染图 / 手绘草图 / 平面图 / 剖面图 / 立面图 / 轴测图 / 分析图 / 实体模型照片 / 拼贴 | 单选（softmax 取最大） |
| **视角** | 外观 / 室内 / 鸟瞰 / 人视 / 细部特写 | 单选 |
| **建筑类型** | 住宅 / 文化建筑 / 高层 / 桥梁与构筑物 / 景观 / 城市街区 / 室内空间 … | 多选（阈值） |
| **材料** | 清水混凝土 / 木材 / 砖 / 玻璃幕墙 / 金属 / 石材 / 膜结构 … | 多选 |
| **氛围/光线** | 白天 / 夜景 / 黄昏 / 雾 / 雪 / 强阴影 / 逆光 | 多选 |
| **风格倾向** | 极简 / 粗野主义 / 参数化 / 古典 / 乡土 / 高技派 … | 多选 |

实现要点：

- 每个标签写成 prompt 模板（如 `"a photograph of a brutalist concrete building"`），中英文双语各编码一次取平均，保证中文检索一致性。
- 标签向量矩阵缓存为一个小 `.npy`，标签表用一个可编辑的 `tags.json` 维护——**用户可自行增删标签词表，改完点"重新打标"即可**，无需重建 FAISS。
- 打标结果写入图像 metadata（`auto_tags: {dimension: [tag, score]}`），与现有 `tags` 手动标签字段并存。

### 4.3 标签的三个用途

1. **结果过滤（最重要）**：搜索结果上方显示当前结果集的标签统计芯片（如 `渲染图 41 | 剖面 12 | 混凝土 28`），点击即过滤。"想要剖面不要效果图"一次点击完成——这是纯向量检索永远做不干净的事。
2. **替代 IntentEngine**：输入框下的候选建议改为"查询词 + 高频共现标签"（如输入 `museum` → 建议 `museum + 清水混凝土`、`museum + 剖面图`），有真实语义，而非文件名词频。
3. **空查询浏览**：不输入任何内容时，首页按"图像类型"维度分组展示图库，方便直接翻找。

## 五、检索方式与反馈机制重构

### 5.1 新增：以图搜图（最高优先级）

- 每张结果卡片增加一个 **"更多类似 ↻"** 按钮：取该图已存的 CLIP 向量直接查 semantic 槽，毫秒级返回。
- 支持**拖入外部图片**作为查询（截图、Pinterest 存图）：现场编码一次 CLIP 即可。
- 支持 **文字 + 参考图混合查询**：`q = α·text_vec + (1-α)·image_vec`，α 用一个滑杆控制（默认 0.5）。这是"我要类似这张但更 X"的标准解法。

### 5.2 反馈机制：从"暗中学权重"改为"显式改查询向量"（Rocchio 相关反馈）

只保留三个显式动作，全部**即时、可见、可解释**：

| 动作 | 交互 | 对检索的影响 |
|---|---|---|
| **更多类似** | 卡片按钮 | 该图 CLIP 向量以正系数并入当前查询向量，立即重排/重搜 |
| **排除 ✕** | 卡片按钮 | 本次会话隐藏该图，且其向量以小负系数并入查询向量（Rocchio 负反馈） |
| **收藏 ★** | 加入 moodboard（保留现有功能） | 纯收藏，不改变检索（避免收藏行为污染当前查询） |

查询向量更新公式（经典 Rocchio）：

```
q' = normalize( q + β·mean(positive_vecs) − γ·mean(negative_vecs) )
β ≈ 0.6, γ ≈ 0.2
```

配套 UI：搜索框旁显示"已叠加 2 正 / 1 负反馈 [清除]"，一键回到原始查询。**每一步用户都知道系统为什么变了，随时可撤销**——这是旧自适应权重最缺的东西。

### 5.3 模态权重：回归简单

- 删除会话自适应与置信度混合，权重只来自设置页滑杆（且现在只剩 semantic/structure/color 三个）。
- 默认场景下用户根本不用碰权重：文字/图片查询走 semantic，画了草图自动叠加 structure，选了色块自动叠加 color——**"提供了哪种输入就启用哪个槽"**，与现有 slot chip 逻辑一致但更自动。

### 5.4 速度体验细节

- 搜索结果**流式/分批渲染**（先出前 30 张再补齐），感知延迟减半。
- CLIP 文本编码结果做 LRU 缓存（同一 query 微调时免重编码）。
- 意图候选面板从"搜索前拦一道"改为"搜索后可选优化"：**回车永远直接出结果**，建议芯片显示在结果上方，点了才二次搜索。

### 5.5 可选升级（第二阶段再考虑）

- CLIP ViT-B/32 → OpenCLIP ViT-L/14 或 SigLIP：建筑材质/风格分辨力明显更强，代价是索引需重建、显存要求更高。建议先完成上述流程改造，若语义质量仍不满意再升级。
- 若升级模型，`indexed_slots` 已有版本字段，可支持双版本并存渐进迁移。

## 六、实施顺序（整合第七、八、九节后的完整版）

| 阶段 | 内容 | 预期收益 |
|---|---|---|
| **P1 减法**（先做，风险低） | 删除 pose/depth/OCR/EXIF/pairwise/hover/skip/diversity/自适应权重/L3L4 导出；意图面板改为不拦截；设置项同步清理（§9.6）；同步修订 MODIFICATION_GUIDE（§9.7） | 代码量约减 1/3，索引提速，交互零打断 |
| **P2 核心检索体验** | "更多类似"按钮、拖图查询、Rocchio 正负反馈叠加与清除（§5）；即输即搜 + 键盘流（§8.2）；事件日志 `feedback_log.jsonl` 开始积累（§7.3，此时只记不用） | 找图速度质变，反馈可见可控 |
| **P3 标签与记忆** | tags.json 词表、零样本批量打标、结果标签过滤芯片、芯片式组合查询、建议下拉、空查询分组首页（§4、§8.2）；查询记忆 + 收藏先验（§7.4-1/2）；moodboard 导出 PDF/PPTX（§9.2） | 精确过滤图纸类型/材料/风格；重找旧图直接置顶 |
| **P4 可选增强** | legacy sketch 槽位清理迁移、CLIP 模型升级、文字+图混合滑杆、相似聚类导航、增量索引监听（§9.1）、个人化适配层（§7.4-4）、跨库搜索（§9.5） | 语义质量与自动化程度进一步提升 |

### 每阶段验收标准

- P1：全流程（导库→索引→搜索→收藏）无弹窗打断；同一查询两次结果完全一致（可复现）。
- P2：纯文本查询从敲字到首屏结果 < 500ms（万张库）；从"看到一张接近的图"到"拿到一屏类似图"≤ 2 秒、1 次点击；3 轮正负反馈内结果明显收敛。
- P3：`剖面图`、`清水混凝土 住宅` 类查询的首屏命中率显著高于纯 CLIP 文本检索；打标 1 万张图 < 1 分钟（纯矩阵运算）；重复上周的查询词，上次收藏的图出现在首行。
- P4：golden query 回归测试集（§9.4）全部通过后才允许模型/逻辑升级合入。

## 七、数据积累反馈：合理的长期学习设计

### 7.1 旧方案为什么不合理

旧的 L3/L4 体系把三件事搅在一起：实时改变检索结果（自适应权重）、采集行为数据（hover/skip/pairwise）、导出研究标注（L3Record）。结果是：数据噪声大（隐式信号不可靠）、学习不可解释（softmax 权重漂移）、且学习直接污染当前会话（结果不可复现）。

### 7.2 核心原则：把"当下反馈"和"长期学习"彻底分层

```
第 1 层  会话内反馈（Rocchio）   —— 即时生效、可见、关掉即清零，永不落盘影响未来
第 2 层  事件日志（append-only） —— 只记录，不参与检索；廉价、无副作用
第 3 层  离线消化（定期批处理）  —— 从日志提炼出带版本号的"学习产物"，可回滚、可清空
第 4 层  检索时应用             —— 学习产物只作为轻量先验/记忆，不改变检索的可预测性
```

### 7.3 只积累高置信度的显式信号

| 记录 | 信号含义 | 不记录 |
|---|---|---|
| 收藏到 moodboard | 强正样本 | hover 时长（噪声） |
| 显式排除 ✕ | 强负样本 | 滚动跳过（噪声） |
| "更多类似"点击且后续有收藏 | 正样本 + 查询链路 | pairwise 强制二选一（样本随机、打断） |
| 查询文本 + 当次展示的 top-N + 最终收藏项 | 完整三元组 (query, positive, shown-but-ignored) | |

存储形式：每个库目录下一个 `feedback_log.jsonl`，每行一条事件（时间戳、会话 id、查询文本与向量哈希、uid、动作）。只追加、体积极小、随时可整体删除（隐私与重置都简单）。

### 7.4 积累数据的四个用途（按实现难度递进）

1. **图像质量先验**：被收藏次数多的图，在同分候选中获得小幅加成（如 `score × (1 + 0.05·log(1+saves))`），带时间衰减、设上限，避免"越推越热"的马太效应。
2. **查询记忆（性价比最高）**：记录"查询 X 时收藏了哪些图"。下次输入语义相近的查询（文本向量余弦 > 0.85），把当年收藏过的图轻量置顶。**"重新找到以前见过的那张图"是找参考图最高频的痛点**，这个功能直接命中。
3. **个人词表校准**：统计"输入后产生了收藏"的查询词，作为 `tags.json` 标签词表和搜索建议的扩充来源——比文件名词频有意义得多，因为这些词经过了"确实找到了图"的验证。
4. **离线评测 + 轻量适配层（数据量够了再做）**：积累的 (query, positive, negative) 三元组首先用作**评测集**——以后升级 CLIP 模型或改检索逻辑，跑一遍就知道收藏图的排名是升是降，改动有据可依。当三元组达到数百条以上，可选训练一个 CLIP 向量之上的线性投影矩阵（triplet loss，CPU 秒级训练完），作为个人化适配层：带版本号、一键停用、效果先过评测集再上线。

### 7.5 必须遵守的约束

- 显式信号 > 隐式信号，宁缺毋滥。
- 学习产物永远离线生成、带版本、可回滚，检索主链路不做在线学习。
- 用户可查看、可一键清空全部积累数据（一个"重置个人化"按钮）。
- 同一会话内，同一查询的结果必须始终一致（先验在会话开始时固定）。

## 八、苹果相册式检索模式：快与便捷的来源

### 8.1 苹果相册快的本质

不是模型更强，而是三件事：**零门槛交互**（没有"搜索按钮"这道门，边输边出结果）、**一切预计算**（场景/人物/地点全部索引时算好，搜索只是查表+过滤）、**零查询也能用**（不搜索时有回忆/分组可翻）。LocalMuse 的 CLIP+FAISS 底子完全支撑得起同样的体验，缺的是交互模式。

### 8.2 检索模式改造清单

| 模式 | 做法 | 说明 |
|---|---|---|
| **即输即搜** | 去掉"必须点 Search"，输入防抖 ~250ms 后自动查询，先渲染 top 30 | CLIP 文本编码 + flat FAISS 在万张级图库是毫秒级，完全够实时；semantic 单槽即可，草图/颜色等重查询仍走显式按钮 |
| **建议下拉** | 输入时下方浮层混合展示：匹配的标签（第四节体系）、历史成功查询（第七节记忆）、文件名/手动标签命中 | 对应苹果的"人物/地点/场景"分类建议；点击即搜 |
| **芯片式组合查询** | 输入的词自动解析为可单独删除的芯片：命中标签词表的变成**过滤芯片**（精确元数据过滤），其余合并为**语义芯片**（CLIP 向量） | "剖面图 混凝土 教堂" → [剖面图·过滤][混凝土·过滤] + "教堂"·语义；比纯向量检索干净，比纯标签检索灵活 |
| **空查询智能首页** | 不输入时展示：最近导入 / 按图像类型分组 / 常被收藏 / 相似聚类入口 | 对应苹果"回忆"；很多时候用户是"翻"而不是"搜" |
| **相似聚类导航** | 索引后对 CLIP 向量跑一次 k-means（离线、秒级），首页展示聚类封面，点击进组 | 无需任何标注的"自动相册" |
| **键盘流** | `/` 聚焦搜索、方向键选图、Enter 放大、`S` 收藏、`X` 排除 | 高频使用下比鼠标快一倍 |

### 8.3 支撑性能的工程要点

- semantic FAISS 索引与缩略图路径表常驻内存（现有架构已基本如此），确认无每次搜索的磁盘 IO。
- 缩略图 HTTP 响应加 `Cache-Control` 长缓存头，翻页/重搜不重传。
- CLIP 文本编码加 LRU 缓存，即输即搜的逐字符前缀不重复编码。
- 结果分批流式渲染（P1 已列）；图库超过 ~10 万张再考虑 IVF/PQ 量化索引，之前 flat 就是最快最准的。

### 8.4 与前述方案的关系

即输即搜是**入口**，标签芯片（第四节）是**收窄**，以图搜图+Rocchio（第五节）是**收敛**，查询记忆（第七节）是**重逢**——四者串起来就是完整的快速找图闭环：

```
敲几个字 → 实时出图 → 点芯片收窄 → 点"更多类似"收敛 → 收藏
下次再找 → 输入相近的词 → 上次收藏的图直接置顶
```

## 九、其他改进建议（工作流与工程层）

### 9.1 索引与库维护

- **增量索引 / 文件夹监听**：给库绑定若干"来源文件夹"，启动时（或用 watchdog 实时）对比文件清单，新图自动入库、已删文件标记失效——建筑学生的图是持续攒的，"存进文件夹就等于入库"能省掉所有手动 import。
- **索引断点续传**：索引中断后记录已完成清单，重启从断点继续，失败文件汇总成可重试列表（现在单张失败只是静默 skip）。
- **批量编码提速**：CLIP 按 batch（如 32 张）编码而非逐张，有 GPU 时索引速度可提升数倍。

### 9.2 建筑工作流集成（moodboard 的下游）

- **Moodboard 导出为 PDF / PPTX 参考图版面**：一键把收藏图排成带出处文件名的 contact sheet，直接用于评图汇报和设计说明——moodboard 目前只能看，导出让它接上真实工作流。
- **复制原图到剪贴板**：卡片右键"复制图片"，直接粘进 PS/InDesign/PPT；配合已有的"打开所在文件夹"覆盖两种取用方式。

### 9.3 浏览与对比体验

- **重复图折叠**：已有 duplicate_group 数据但结果里仍平铺展示；同组折叠为一张 + 角标 `+N`，点开看全部版本，结果页更干净。
- **Lightbox 双图对比**：放大视图中支持选两张并排对比（快捷键 `C`）——这是旧 pairwise 弹窗真正合理的形态：**用户主动发起的对比**，而非系统强制的二选一。
- **大库虚拟滚动**：结果网格改虚拟滚动/瀑布流，几千张结果也不卡（配合 §5.4 的分批渲染）。

### 9.4 质量保障

- **Golden query 回归测试集**：在现有 `test_search.py` 基础上固化一组"查询→期望图必须进前 N"的用例（可直接由 §7.4 的三元组生成），任何检索逻辑或模型改动先跑测试再合入——保证"改进"不是感觉，是可测量的。
- **中文链路验证**：确认多语言文本编码器（mclip）在当前库真实生效（`indexed_slots` 里能查到 `clip-vit-b32-mclip`），并把"中文查询 ≈ 对应英文查询结果一致性"纳入 golden query 测试。

### 9.5 多库支持（低优先级，可不做）

- 学生常按课程/项目分库。可选提供"跨库统一搜索"：各库索引独立不动，查询时并行查多库、结果合并标注来源。仅当确实形成多库习惯后再做。

### 9.6 配置与代码清理

- 设置页与 `settings.json` 同步删除失效开关（pose/depth/ocr enabled 与权重、pairwise 相关常量），避免"设置里有但没作用"的僵尸选项。
- `session_tracker.py` 精简后只剩 moodboard + Rocchio 状态 + 事件日志写入，预计从 ~640 行缩到 ~150 行。

### 9.7 文档同步

- 本方案实施时**同步修订 `LOCALMUSE_MODIFICATION_GUIDE.md`**：其中"意图候选必须来自库内词频""单一 Search 按钮为唯一触发"等规则将被本方案取代（见 §10 与 §11.2），过期规则不删会误导后续开发。

## 十一、全框架一致性自检

新增建议后对整份方案做了一次交叉检查，结论与已做的修正：

### 11.1 流程闭环完整性 ✓

入口（即输即搜 §8.2）→ 收窄（标签芯片 §4.3/§8.2）→ 收敛（以图搜图 + Rocchio §5.1/5.2）→ 落袋（moodboard → PDF/PPTX 导出 §9.2）→ 重逢（查询记忆 §7.4）。每个环节有且只有一个机制负责，无重复无缺口。

|---|---|

### 11.3 依赖关系检查 ✓

- 事件日志（P2 开始记）先于查询记忆/先验（P3 使用）与适配层/评测集（P4 使用）——数据先行，正确。

### 11.4 原则一致性检查 ✓

全方案贯穿四条不变原则，逐条核对无违反：**结果可复现**（会话内同查询同结果；先验会话开始时固定）、**反馈可见可撤销**（Rocchio 状态条、个人化重置按钮）、**学习与检索分离**（在线只用离线产物）、**显式信号优先**（唯一保留的"隐式"数据只是曝光记录，仅用于离线评测，不影响排序）。

### 11.5 残留风险提示

- 即输即搜在**纯 CPU + 大库**场景下文本编码可能达数百毫秒：预案是防抖加长至 400ms 并保留回车立即触发，不影响架构。
- 零样本打标的阈值需按库实测校准（不同图库分数分布不同）：tags.json 中为每维度留 threshold 字段，打标工具输出分数分布直方图辅助定阈值。
- Rocchio 负反馈系数 γ 过大会把结果推向反方向：默认从 0.2 起步，排除主要靠"会话内隐藏"兜底，向量负反馈只是辅助。
